RABA Prototype

Responsibility Review Panel

A minimal interface prototype showing how RABA separates baseline compliance from runtime governance before an AI-supported action becomes a business effect.

1. Action Statement

AI-supported action
Send a customer-facing email drafted by an AI support assistant.

The message contains customer account context and proposes a compensation response.

Expected effect
External communication to customer
Affected system
Support email / CRM workflow

2. Runtime Governance

Boundary: external customer communication Risk: medium Uncertainty: known and bounded

RECOMMEND
AUTHORIZED
EXECUTED
ESCALATE

Current question: can this concrete action move from RECOMMEND to AUTHORIZED, or should it be escalated / deferred / returned for more context?

3. Responsible Subject

Acting role
Customer Support Lead
Authority
Valid for medium-risk customer communication
Continuity
Responsible until customer response is reviewed
Exit path
Escalate / request more context / defer

Readiness is declared through accountable role-based action, not inferred through behavioral monitoring.

Baseline Compliance

Allowed with conditions

Static compliance sets the floor. Runtime governance controls the crossing.

Sources
GDPR / EU AI Act / Internal Customer Communication Policy
Required controls
Human review, Decision Log, escalation path

Human Readiness Check

I understand what action is being authorized.
I understand the boundary and possible external effect.
The risk is inside my role authority.
I know the escalation path if uncertainty increases.

Decision Options

Authorize is not the only normal path. Responsible refusal and escalation are valid governance outcomes.
