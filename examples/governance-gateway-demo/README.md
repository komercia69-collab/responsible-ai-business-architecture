# Minimal Governance Gateway Demo

## Purpose

This example turns the Responsible AI Business Architecture concepts into a small executable proof layer.

It demonstrates how an AI agent action can be checked before it becomes operational action.

The goal is not to build a production-ready gateway.

The goal is to show the basic logic of:

```text
AI agent action
        ↓
Governance policy
        ↓
Governance Gateway checks
        ↓
ALLOW / BLOCK / REQUIRE_APPROVAL / ESCALATE
        ↓
Decision log
```

---

## Scenario

A customer support AI agent wants to perform or recommend actions such as:

- classify a support ticket;
- draft a customer reply;
- offer a refund;
- update a CRM record;
- close a complaint;
- send a legal or medical statement.

The Governance Gateway decides whether the action is allowed, blocked, requires human approval, or must be escalated.

## How to Run

From the repository root:

```bash
node examples/governance-gateway-demo/governance-gateway.js
```

No external dependencies are required.

---

## How to Test

From the repository root:

```bash
node --test examples/governance-gateway-demo/governance-gateway.test.js
```

The tests verify that the gateway can return the core decision types:

- `ALLOW`
- `BLOCK`
- `REQUIRE_APPROVAL`
- `ESCALATE`

They also check important governance paths such as forbidden categories, refund thresholds, irreversible actions and high-risk categories.

---

## What This Demonstrates

This demo shows how abstract governance concepts become operational checks.

| RABA concept | Demo implementation |
|---|---|
| AI Action Boundary Mapping | Action type, category, customer impact and execution mode are checked |
| AI Governance Gateway | Central function evaluates policy before action proceeds |
| Runtime Governance | Each action is evaluated at runtime |
| Human Approval Point | Certain actions return `REQUIRE_APPROVAL` |
| Corrective Governance | High-risk or irreversible actions return `ESCALATE` |
| Governance Nervous System | Decisions are logged with reasons and timestamps |

---

## Decision Types

| Decision | Meaning |
|---|---|
| `ALLOW` | The action is inside the approved boundary |
| `BLOCK` | The action is not allowed by policy |
| `REQUIRE_APPROVAL` | A human must approve before execution |
| `ESCALATE` | The action requires higher-level review or specialist handling |

## Example

Input action:

```json
{
  "id": "action-003",
  "agent": "customer-support-agent",
  "actionType": "offer_refund",
  "executionMode": "execute",
  "amount": 120,
  "customerImpact": true,
  "reversible": true,
  "category": "refund",
  "description": "Offer a refund of 120 EUR to a dissatisfied customer."
}
```

Policy rule:

```json
{
  "maxAutonomousRefundAmount": 50,
  "maxRefundWithManagerApproval": 250
}
```

Gateway decision:

```text
REQUIRE_APPROVAL
```

Reason:

```text
Refund exceeds autonomous threshold and requires human approval.
```

## Why This Matters

A prompt is not a governance architecture.

A policy document is not enough if it does not affect runtime behavior.

This demo shows the smallest useful bridge between:

```text
Responsible AI principle
        ↓
Governance policy
        ↓
Executable control point
        ↓
Auditable decision log
```

Tests add another layer:

```text
Governance policy
        ↓
Executable control point
        ↓
Expected decisions are verified
```

---

## Limitations

This is a minimal educational demo.

It does not include:

- authentication;
- authorization;
- real tool integration;
- persistent database logging;
- cryptographic audit logs;
- real CRM connection;
- legal compliance checks;
- production security hardening;
- human approval UI.

A production implementation would require proper engineering, security, privacy, compliance and domain review.

---

## Key Principle

> Before AI acts, define where AI is allowed to act.

> Before AI executes, create a gateway.

> Before autonomy scales, make responsibility executable.
