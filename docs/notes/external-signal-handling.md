# External Signal Handling

## Status

This document is a working note.

It is non-canonical. It is not approval, validation, endorsement, adoption evidence, partnership evidence, public positioning, compliance evidence, or certification evidence.

## Core rule

External signal does not equal validation.

Interesting idea does not equal canon.

Contact does not equal partnership.

AI agreement does not equal approval.

Repository presence does not equal architectural approval.

## Purpose

RABA may receive many useful external signals:

- LinkedIn posts;
- expert comments;
- accepted contact requests;
- AI reviews;
- practitioner observations;
- provider statements;
- public articles;
- screenshots;
- informal discussions;
- repeated patterns noticed during project work.

The purpose of external signal handling is to preserve useful observations without overclaiming their meaning.

## Why this matters

External signals can be valuable.

They may show that a problem is real, repeated, or professionally relevant.

But they can also create a dangerous illusion of confirmation.

A signal may feel like evidence before it has been verified.

A contact may feel like support before the person has reviewed anything.

A model agreement may feel like validation before a human authority has decided anything.

RABA must prevent this confusion.

## Signal classification

External materials should be classified before use.

Possible classifications:

- raw signal;
- external signal;
- working thought;
- review input;
- contact signal;
- market signal;
- technical signal;
- governance signal;
- parked;
- needs verification;
- rejected or archived.

## Required boundary language

Every external signal should preserve the following boundary:

```text
This signal does not imply validation, adoption, endorsement, partnership, canon, compliance readiness, certification readiness, or public-use approval.
```

## Human Owner decision required

Human Owner confirmation is required before an external signal becomes:

- a canonicalization candidate;
- public positioning;
- outreach basis;
- provider-facing material;
- commercial claim;
- partnership claim;
- adoption claim;
- validation claim;
- compliance or certification claim;
- GitHub issue;
- pull request;
- canon.

## Safe use

External signals may be used safely as:

- review aids;
- thought seeds;
- problem examples;
- risk indicators;
- language stress-tests;
- comparison material;
- prompts for later investigation.

They should not be used as proof that RABA is accepted, validated, adopted, or externally endorsed.

## Contact-specific boundary

For external contacts:

```text
Connection does not mean partnership.
Reply does not mean validation.
Interest does not mean adoption.
Agreement does not mean approval.
```

A person should not be described as a RABA reviewer, supporter, partner, validator, or adopter unless they explicitly agree to that role.

## AI-review boundary

For AI-generated reviews:

```text
AI review is review aid only.
Multi-AI agreement is not Human Owner approval.
AI recommendation is not canon.
AI confidence is not architectural authority.
```

## GitHub boundary

External signals should not automatically become GitHub material.

Suggested sequence:

```text
Inbox first.
Selection second.
Human Owner decision third.
GitHub action last.
```

Drive or inbox capture does not authorize repository changes.
