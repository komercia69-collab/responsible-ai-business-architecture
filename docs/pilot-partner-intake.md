# Pilot Partner Intake

This document provides a short intake structure for early RABA pilot conversations.

It is designed to help identify one narrow workflow where AI-supported output may become business action and where RABA can add visible governance.

---

## Purpose

The goal of intake is not to assess the entire organization.

The goal is to identify one workflow suitable for a minimal RABA pilot.

---

## Basic Context

```text
Organization / team:
Product or workflow:
Primary contact:
Roles involved:
Current AI system or integration:
```

---

## Workflow Questions

1. What AI-supported workflow are you currently using or planning?
2. What does the AI system currently do?
3. Does AI output stay internal, or can it influence customer-facing, financial, legal, operational, or system-of-record actions?
4. Where does AI assistance become business action?
5. Who currently owns that transition?

---

## Action Boundary Questions

1. What is the AI allowed to do today?
2. What is it not allowed to do?
3. Is that boundary documented, coded, or only understood informally?
4. Can the AI read sensitive or high-impact context?
5. Can the AI write, send, submit, update, close, approve, trigger, or delete anything?

---

## Approval Questions

1. Which actions require human review today?
2. Who can approve them?
3. Is approval explicit or implied by user click / workflow continuation?
4. Can approval expire?
5. Are there cases where re-approval is required?

---

## Logging and Audit Questions

1. What is logged today?
2. Is the log technical only, or does it explain the business decision?
3. Can you answer who allowed the AI-supported action to progress?
4. Can you link the final action to the recommendation and approval that preceded it?
5. Would the current record satisfy an internal review or audit question?

---

## Escalation Questions

1. What happens when the AI output is uncertain, high-impact, or outside a clear boundary?
2. Who receives the escalation?
3. Is escalation tracked?
4. Can escalation return the action to review, approve it, reject it, or close it with no action?
5. Are overrides visible and recorded?

---

## Pilot Fit Questions

1. What is the smallest workflow where this would be useful?
2. Which RABA primitives would help most?

```text
Action Boundary
Approval States
Governance Gateway
Decision Log
Escalation Path
Responsibility Event Stream
```

3. What would make a first pilot valuable?
4. What would make it too heavy?
5. Who should join a 60–90 minute workflow-mapping session?

---

```text
Workflow:
AI-supported action:

```

---

## Phase 1 Boundary

Phase 1 focuses on governing action progression after an AI-supported action request has been formed.

Input Integrity Governance, production identity, full UI, signing, monitoring and enterprise rollout are out of scope for the first pilot unless explicitly selected as a separate future track.
